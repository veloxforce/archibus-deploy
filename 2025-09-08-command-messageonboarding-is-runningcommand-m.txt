‚è∫ Architecture Decision Record: Configure LibreChat Interface for End-User Simplicity

  Status: AcceptedDate: 2025-09-08Deciders: Platform Architecture Team

  Context and Problem Statement

  LibreChat's default interface exposes numerous technical features designed for power users and
  developers. When deploying LibreChat for non-technical end users (customer service, facility
  management, healthcare staff), these features create confusion, increase training costs, and reduce
   adoption rates. A streamlined interface is needed that presents only essential chat functionality.

  Decision Drivers

  - User Experience: Non-technical users need simple, intuitive interfaces
  - Training Costs: Complex UIs require extensive onboarding
  - Support Burden: Technical options generate unnecessary support tickets
  - Professional Appearance: Client-facing deployments need clean presentation
  - Use Case Focus: Most deployments serve specific business functions, not general AI exploration

  Considered Options

  1. Full Feature Interface - Keep all LibreChat features visible
  2. Selective Hiding - Disable only the most technical features
  3. Minimal Interface - Hide all non-essential elements for pure chat experience
  4. Role-Based Interface - Different configurations per user type

  Decision Outcome

  Chosen option: "Minimal Interface" - systematically disable all technical UI elements for end-user
  deployments.

  Configure each interface option based on end-user needs rather than power-user capabilities.

  Interface Options Reference

  Core UI Elements

  | Option      | Purpose                                      | End-User Setting
    | Power-User Setting                      |
  |-------------|----------------------------------------------|-------------------------------------
  --|-----------------------------------------|
  | modelSelect | Shows AI model dropdown selector             | false - Users shouldn't switch
  models | true - Developers need model comparison |
  | parameters  | Temperature, token limits, sampling controls | false - Too technical for end users
    | true - Fine-tuning for specific outputs |
  | sidePanel   | Agent Builder and configuration panel        | false - No agent creation needed
    | true - Custom agent development         |
  | presets     | Saved configuration sets                     | false - Adds complexity
    | true - Workflow optimization            |
  | prompts     | Prompt library and templates                 | false - Predefined via system
    | true - Prompt engineering               |
  | multiConvo  | Multiple conversation tabs                   | false - Single focus preferred
    | true - Parallel experiments             |
  | agents      | Agent selection and management               | false - Single-purpose deployment
    | true - Multi-agent workflows            |
  | bookmarks   | Bookmark conversations                       | false - Depends on use case
    | true - Research and reference           |

  Advanced Features

  | Option        | Purpose                       | End-User Setting                | Power-User
  Setting                 |
  |---------------|-------------------------------|---------------------------------|----------------
  --------------------|
  | endpointsMenu | API endpoint/server selection | false - Backend predetermined   | true - Testing
  different providers |
  | webSearch     | Web search integration button | false - Unless core to use case | true - Research
   augmentation       |
  | fileSearch    | Search through uploaded files | false - Simplifies interface    | true - Document
   analysis           |
  | runCode       | Code execution capabilities   | false - Security and complexity | true -
  Development workflows       |
  | fileCitations | Shows source file references  | false - Technical detail        | true -
  Verification needed         |

  Professional Customization

  | Option         | Purpose                       | Configuration Guidance
                 |
  |----------------|-------------------------------|-------------------------------------------------
  ---------------|
  | customWelcome  | Personalized greeting message | Always set - brand-appropriate welcome improves
  engagement     |
  | privacyPolicy  | Privacy policy link           | Configure for compliance - required for
  enterprise deployments |
  | termsOfService | Terms acceptance flow         | Set modalAcceptance: true for first-use
  agreement              |
  | marketplace    | Plugin/extension marketplace  | false - Avoid feature creep for end users
                 |
  | peoplePicker   | User/group/role selection     | Configure based on collaboration needs
                 |

  Configuration Examples

  Minimal End-User Configuration

  interface:
    customWelcome: "Welcome! How can I assist you today?"
    # Hide all technical elements
    modelSelect: false
    parameters: false
    sidePanel: false
    presets: false
    prompts: false

    agents: false
    bookmarks: false
    endpointsMenu: false

  Power-User Development Configuration

    customWelcome: "LibreChat Development Environment"
    # Enable all capabilities
    modelSelect: true
    parameters: true
    sidePanel: true
    presets: true
    prompts: true

    endpointsMenu: true
    webSearch: true

    fileCitations: true

  Consequences

  Positive

  - ‚úÖ Reduced Cognitive Load: Users focus on conversation, not configuration
  - ‚úÖ Faster Adoption: Minimal training required for basic chat interaction
  - ‚úÖ Lower Support Costs: Fewer "what does this button do?" tickets
  - ‚úÖ Professional Appearance: Clean interface suitable for client demonstrations
  - ‚úÖ Predictable Behavior: No accidental model switches or parameter changes

  Negative

  - ‚ùå Limited Flexibility: Users cannot optimize for specific tasks
  - ‚ùå No Experimentation: Cannot test different models or approaches
  - ‚ùå Hidden Capabilities: Users unaware of system's full potential
  - ‚ùå One-Size-Fits-All: Same limitations for all users regardless of skill

  Neutral

  - ‚ö™ Reversible Changes: All settings modifiable via configuration file
  - ‚ö™ Per-Deployment Choice: Each installation can choose appropriate level
  - ‚ö™ Documentation Needed: Must explain to stakeholders why features are hidden

  Implementation Guidance

  1. Assess User Base: Determine technical proficiency of primary users
  2. Identify Core Workflows: Enable only features essential to business function
  3. Start Minimal: Begin with everything disabled, enable as needed
  4. Monitor Feedback: Track user requests for hidden features
  5. Document Decisions: Explain to users why interface is simplified

  Notes

  - This ADR applies to any LibreChat deployment prioritizing end-user simplicity
  - Power-user deployments should use opposite settings
  - Consider role-based configurations if mixed user base exists
  - Review quarterly as LibreChat adds new interface options
